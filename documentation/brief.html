UFCEUS-20-2 Web Programming
Assessed Coursework 2011/12
Hand-in Deadline : Thursday 22nd March 2.00pm

Learning objectives

·     Build a web application that supports transactions and persistence.

  • Use a range of web oriented software architecture and design principles.
  • Refactor a lightweight PHP-based web framework.
  • Understand and implement both static and dynamic software patterns.
  • Model and implement persistence and transactions via sessions, cookies and DBMS.
  • Understand & apply some security and integrity principles.

Requirements

You are required to select a domain, from the list provided below, and then to design, build and implement an application by using, extending and refactoring the provided PHP framework.  Once completed, the application must be made available for viewing and use within the UWE / FET / CSCT-EDM domain.  For database functionality, your application must make use of an UWE based MySQL server.

Deliverables

System Design, Models, Implementation and Documentation: D1, D2 and D3 (75%)

  • A conceptual Entity Relationship model and a MySQL instantiation of the domain data.
  • A user registration and authentication component.
  • An account and cart management component.
  • A component to consume the RESTful card authentication service.
  • A search component.

Your implementation must make use of the provided PHP framework (see Resources in this document) but you can modify, extend and refactor it as much as you require.  Further use of: PHP, MySQL, XHTML, XML, Javascript and CSS is expected.  You are free to make use of any open source resources such as Javascript and PHP libraries, CSS skins, etc. but these must be properly acknowledged.

Once completed – the base page of your application (index.php) must be available at the following URL:

http://www.cems.uwe.ac.uk/[~yourusername]/wp/assignment/index.php

No other URL will be accepted. Note that if at the time of marking, the application is not found at this URL – 25% of the overall marks will be deducted.

Reusability Analysis: D4 (25%)

Your system should support data entry by users and should facilitate some retrieval of stored data; an interface needs to be applied to the system to facilitate these actions. You need to explain the design of the interface(s) and show how this design has been informed by consideration of design theories and of the perspective of your intended user ‘audience’.

Having presented a reasoned and evaluated case for the design of your system, you should now explain how the system itself and/or any elements that you have used in it (or in the processes used in building it) could be usefully employed elsewhere and why that might be valuable (in terms of promoting and exploiting the reuse of patterns and components and improving productivity and production of quality products).

Marking Criteria

D1 - Working Application (25%)

  • Accessible within UWE site as shown by a presentation by you to tutors present
  • XHTML is valid
  • User Interface allows required functions to run in a simple interaction

D2 - Design & Implementation (25%)

  • Overall systems architecture, which includes consideration of security
  • Database design and implementation (including an adequate dataset)
  • Implementation of the required functions and components
  • Adoption, use and consistent extension of the configuration file

D3 - System Documentation (25%)

  • E-R Models and SQL DDL & DML listings
  • program and data listings
  • mark-up listings &, CSS listings
  • system design considerations ensuring data and service security
  • all listings are supplied, well-formed and readable
  • all scripts include appropriate comments
  • code is readable and well-structured, using configuration, functions and library files as appropriate.
  • code and images, including functions and sections of code which have not been written by you must be properly acknowledged

D4 - Reusability Analysis (25%)

  • The overall system is carefully critiqued and categorised
  • The system as a whole is interpreted in the context of systems elsewhere
  • A variety of elements of the development are itemised and their value in promoting reuse is clearly rationalised and presented.
  • The notion of reuse and improvement in productivity and production of quality products is clearly analysed using the web application and its production processes.

Overall assessment criteria

Models, Services & Components D1+D2+D3 (75%)

0 - 39 (fail) for an attempt which does not work, is documented only by listings of dubious provenance and which fails to show any learning in the technologies required.

40-49 for an attempt which has a few basic functions and remains poorly documented. Parts of the application may not work, but this can be seen to be because of a simple problem and there is good work in evidence despite the failure to work as a whole.

50-59 for a basically functioning system produced, with some documentation.

60-69 for a system with most functions working, a basic interface and good documentation.

70+ for a fully functioning, well-documented system.

80+ for a system of sufficient quality to be used to promote the module in Open Days.

Analysis D4 (25%)

0 - 34 poor understanding or justification by reference to standards and further uses.

35 - 49 minimal understanding with description of standards and theory but little critical interpretation of explanation of the value to the particular system produced

50- 59 straightforward application of the principles to the analysis but little breadth of research or application of theory

60 - 69 interesting analysis and justification but lacks wide exploration or in depth analysis of the value of the produced system to a variety of situations

70 + insightful report, clearly ties together theory and principles to the system as produced and shows a wide appreciation of applications in a wide range of situations
